# 智能食谱助手需求文档

## 项目目标

基于免费公开 API 构建智能食谱助手。用户通过 Chat 描述想吃什么，Agent 自主调用工具搜索菜谱和饮品，并给出结构化推荐。

## 题目来源

对应面试题目 A：智能食谱助手。

## 用户场景

- 用户输入想吃的菜、食材、地区菜系或饮食偏好。
- 用户要求推荐一顿晚餐。
- 用户查看上一次结果中的某个菜谱详情。
- 用户希望获得主菜和配饮组合。

## 功能需求

### AI Agent

- 使用 OpenRouter API 调用大语言模型。
- 兼容 OpenAI 配置。
- 支持多轮对话。
- Agent 能根据自然语言输入自主决定调用工具。
- 食谱、饮品相关问题必须先调用工具获取真实公开数据。
- 不凭空编造菜谱或饮品。

### Tool / Function

必须实现并已实现：

- `search_meals`：按名称、食材、分类、地区或随机搜索菜谱。
- `get_meal_detail`：按菜谱 ID 获取食材清单、用量、烹饪步骤、图片、分类和地区。
- `search_cocktails`：按名称、食材或随机搜索饮品/鸡尾酒。

### Web 界面

- Chat 对话界面。
- 用户输入后流式展示 Agent 回复。
- 菜谱和饮品以卡片结构化展示，包含图片、名称、来源、分类、地区或杯型、部分食材。
- 展示工具调用链路。
- 支持快捷示例入口。

### 数据来源

- TheMealDB：菜谱搜索、食材筛选、分类筛选、地区筛选、详情查询、随机推荐。
- TheCocktailDB：饮品搜索、食材筛选、详情查询、随机推荐。

## 加分项实现

- 餐食搭配推荐：主菜 + 配饮。
- 饮食偏好过滤：素食、地区菜系、食材等。
- Agent 工具调用过程可视化。
- SSE 流式输出。
- 错误处理：模型鉴权、公开 API 超时、工具参数异常、SSE 解析异常。

## 非功能需求

- API Key 只在服务端读取。
- 本地优先读取 `.env.local` / `.env`。
- 公开 API 请求设置超时。
- 前端不展示原始 JSON。
- 项目可通过 `npm run build` 构建。

## 环境变量

OpenRouter：

```txt
OPENROUTER_API_KEY=你的 OpenRouter API Key
OPENROUTER_MODEL=openai/gpt-oss-120b:free
OPENROUTER_BASE_URL=https://openrouter.ai/api/v1
```

OpenAI：

```txt
OPENAI_API_KEY=你的 OpenAI API Key
OPENAI_MODEL=gpt-5.5
OPENAI_BASE_URL=https://api.openai.com/v1
```

优先级：配置了 `OPENROUTER_API_KEY` 时走 OpenRouter；否则走 OpenAI。

## 验收标准

- 用户可在页面发起对话并收到回复。
- Agent 能触发至少 3 个工具中的合适工具。
- 菜谱和饮品以结构化卡片展示。
- 工具调用轨迹可见。
- 支持多轮上下文，例如“看看第一个菜谱详情”。
- 测试、lint、build 可通过。
